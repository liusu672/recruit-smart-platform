# Recruit Smart 前端框架与开发规范

> 更新时间：2026-07-14  
> 适用目录：`frontend/`

> [!IMPORTANT]
> 所有前端页面、组件、交互、颜色、间距、圆角、字体、阴影和 AI 界面必须遵守同目录下的 `design.md`。`design.md` 是视觉与交互的唯一事实来源。框架或组件库默认样式与其冲突时，必须覆盖默认样式，禁止自行引入第二主色、另一套圆角体系或通用后台模板风格。

## 1. 当前状态

前端工程已经完成初始化，不再处于“仅完成技术选型”的阶段。当前是一个基于 Vue 3 的桌面端 SPA，已具备：

- Vite、TypeScript、Vue Router、Pinia 和 Element Plus 基础框架。
- Axios 统一请求客户端、JWT 请求头注入和后端 `Result<T>` 解包。
- 登录会话持久化、角色校验、路由守卫和按角色过滤的侧栏。
- 候选人登录、管理端角色登录和候选人注册页面。
- TanStack Vue Query 全局配置。
- AI SSE 请求客户端和 AI 审批界面示例。
- 工作台页面，以及候选人、招聘流程、面试任务等路由占位页面。
- Vitest 单元测试、ESLint、Prettier 和 `vue-tsc` 检查链路。

当前仍需注意：

- 候选人注册页已完成前端表单和校验，但后端尚未提供注册接口，提交不会创建账号。
- 候选人验证码登录只完成界面，当前后端登录接口仅支持账号密码。
- 候选人、招聘流程和面试任务页面目前是框架占位，尚未接入真实业务数据。
- TanStack Table、VeeValidate、Zod、ECharts 和 Vue ECharts 已安装，但当前业务源码尚未实际使用。

新增业务功能时优先沿用现有边界：

- 路由页面放在 `views/<domain>/`。
- 可复用业务组件放在 `components/<domain>/`。
- 页面状态与交互逻辑放在 `composables/`。
- 静态文案、菜单和选项配置放在 `config/`。
- HTTP 和响应适配只放在 `api/`。
- 跨模块类型放在 `types/`。
- 只有出现多个完整业务模块后，才考虑新增 `features/`，不要提前迁移现有结构。

## 4. 应用启动流程

入口文件为 `src/main.ts`，启动顺序如下：

1. 引入 Element Plus 默认样式和 Recruit Smart 全局样式。
2. 创建 Vue 应用和 Pinia。
3. 注册 TanStack Vue Query。
4. 注册 Vue Router。
5. 注册 Element Plus。
6. 挂载到 `#app`。

`src/App.vue` 只承载根级 `<RouterView />`，登录页、注册页和登录后的应用框架由路由决定。

## 5. 路由与角色权限

当前路由表：

| 路径 | 页面 | 允许角色 | 状态 |
|---|---|---|---|
| `/login` | 登录页 | 公共 | 已实现 |
| `/register` | 候选人注册页 | 公共 | 前端已实现，后端注册接口未接入 |
| `/` | 工作台 | 全部角色 | 已实现基础指标和任务列表 |
| `/candidates` | 候选人 | `HR` | 占位页面 |
| `/pipeline` | 招聘流程 | `HR`、`INTERVIEWER` | 占位页面 |
| `/interviews` | 面试任务 | `HR`、`INTERVIEWER` | 占位页面 |
| `/ai-approvals` | AI 审批 | `ADMIN`、`HR` | 已实现静态审批结构示例 |

全局守卫行为：

1. `meta.public` 路由直接放行。
2. 未登录用户访问受保护路由时跳转 `/login`，并通过 `redirect` 保存目标地址。
3. 已登录但角色不匹配时返回工作台。
4. 侧栏根据当前角色过滤入口。

前端权限只改善用户体验，不能代替后端鉴权。后端仍必须校验 Token、角色和数据归属。

## 6. 登录与会话

登录请求位于 `src/api/auth.ts`：

```text
POST /api/auth/login
```

开发环境中，Vite 会将 `/api` 代理到 `http://localhost:8080`，并在转发时移除 `/api` 前缀，因此后端实际接收 `/auth/login`。

登录响应由 API 层转换为统一的 `LoginPayload`。仅接受以下角色：

- `ADMIN`
- `HR`
- `INTERVIEWER`
- `CANDIDATE`

后端返回空角色或未知角色时，前端抛出 `ApiError(403)`，禁止默认授予 HR 权限。

会话由 `src/stores/session.ts` 管理，当前使用 `sessionStorage`：

```text
recruit-smart-token
recruit-smart-token-type
recruit-smart-user
```

Axios 请求拦截器自动添加：

```http
Authorization: <tokenType> <token>
```

收到 HTTP `401` 时会清理本地会话。当前响应拦截器只负责清理，不直接执行路由跳转；需要跳转时应在统一认证失效流程中扩展，不能在业务页面重复处理。

## 7. API 与响应适配

所有 REST 请求统一使用 `src/api/http.ts`，默认配置为：

```text
baseURL: /api
timeout: 15000ms
```

后端统一响应类型：

```ts
export interface Result<T> {
  code: number
  message: string
  data: T | null
}
```

`unwrapResult<T>` 将 `code` 为 `0` 或 `200` 视为成功，其他业务码转换为 `ApiError`。页面组件不得直接处理 Axios 原始响应壳。

前端分页内部类型已经定义，但尚未接入实际列表接口：

```ts
export interface PagedData<T> {
  items: T[]
  page: number
  pageSize: number
  total: number
}
```

后端分页契约明确后，只能在 API 适配层转换为 `PagedData<T>`，页面不得猜测后端字段名。

## 8. 状态管理边界

### Pinia

当前 Pinia 只存储登录会话。后续可以存储跨页面、纯客户端且需要持久存在的 UI 状态，但不得把候选人列表、职位列表或分页结果复制进 Pinia。

### TanStack Vue Query

Vue Query 已在 `src/main.ts` 注册，默认策略为：

- 查询数据 `30s` 内视为新鲜。
- 查询失败重试 `1` 次。
- Mutation 不自动重试。
- 窗口重新聚焦时不自动刷新。

当前没有业务页面调用 `useQuery` 或 `useMutation`。新增服务端数据请求时，应使用 Vue Query 管理缓存、加载、错误和失效状态。

## 9. AI SSE 与人工决策

`src/api/aiStream.ts` 使用 `@microsoft/fetch-event-source` 发起 POST SSE 请求，支持：

- JWT 请求头。
- JSON 请求体。
- `AbortSignal` 停止生成。
- 类型化事件回调。
- 统一错误回调。

事件类型定义在 `src/types/ai.ts`，包括 `meta`、`delta`、`source`、`tool`、`approval`、`done` 和 `error`。

AI 只能生成摘要、建议、草稿和风险提示。改变候选人阶段、发送 Offer、拒绝候选人、安排面试或转员工档案等操作，必须由业务服务执行，并经过用户明确确认。

## 10. 样式与设计实现

样式入口为 `src/styles/index.scss`：

- `tokens.scss` 定义颜色、间距、圆角、阴影、动效和布局尺寸。
- `element-plus.scss` 将 Element Plus 映射到 Recruit Smart Token。
- `login.scss` 和 `register.scss` 只服务认证页面。

当前关键约束：

- 主色为 `#3183D8`。
- 桌面端最低完整宽度为 `1280px`，全局 `html` 设置了 `min-width: 1280px`。
- 页面高度优先使用 `100dvh`，避免浏览器工具栏造成溢出。
- 动效必须适配 `prefers-reduced-motion`。
- 新增视觉值前必须先检查 `design.md` 和 `tokens.scss`。
- Element Plus 只提供控件能力，不决定产品视觉。

当前包含两组测试文件，共 6 项测试：

- `session.spec.ts`：会话写入、认证状态和清理。
- `auth.spec.ts`：四种合法角色和未知角色拒绝逻辑。

测试环境使用 `jsdom`。后续优先覆盖：

- 路由角色守卫。
- `unwrapResult<T>` 成功与错误映射。
- 登录表单提交和错误状态。

- 业务状态变更前的人工确认规则。

## 13. 新功能落地规则

1. 开发前读取 `design.md` 和本文档。
2. 先确认对应后端接口和字段，不在页面中猜测契约。
3. API 调用放在 `api/`，响应适配不得散落在组件中。
4. 服务端状态优先使用 Vue Query，纯客户端跨页状态才使用 Pinia。
5. 路由通过 `meta.roles` 声明角色，侧栏与路由守卫保持一致。
6. 页面组件只负责组合，复杂交互放入 `composables/`。
7. 表单必须覆盖字段错误、表单错误和重复提交保护。
8. 页面必须考虑加载、空数据、错误、无权限和成功状态。
9. AI 建议必须展示来源，并保留人工确认入口。
10. 完成后运行类型检查、Lint、测试和生产构建。

## 14. 禁止事项

- 禁止绕过 `/api` 和统一 HTTP 客户端直接访问业务服务。
- 禁止页面组件自行读写 JWT。
- 禁止把服务端列表缓存复制到 Pinia。
- 禁止把尚未接入的依赖描述成已完成业务能力。
- 禁止未知角色回退为 HR 或其他高权限角色。
- 禁止 AI 未经用户确认直接改变招聘业务状态。
- 禁止为了 Element Plus 默认样式偏离 `design.md`。
- 禁止在未更新本文档的情况下更换核心框架、路由、状态管理或认证策略。
