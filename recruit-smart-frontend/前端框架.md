# Recruit Smart 前端框架与开发规范

> 更新时间：2026-07-18
> 适用目录：`recruit-smart-frontend/`

> [!IMPORTANT]
> 所有前端页面、组件、交互、颜色、间距、圆角、字体、阴影和 AI 界面必须遵守同目录下的 `design.md`。`design.md` 是视觉与交互的唯一事实来源。框架或组件库默认样式与其冲突时，必须覆盖默认样式，禁止自行引入第二主色、另一套圆角体系或通用后台模板风格。

## 1. 当前状态

前端工程已经完成初始化，不再处于“仅完成技术选型”的阶段。当前是一个基于 Vue 3 的桌面端 SPA，已具备：

- Vite、TypeScript、Vue Router、Pinia 和 Element Plus 基础框架。
- Axios 统一请求客户端、JWT 请求头注入和后端 `Result<T>` 解包。
- 登录会话持久化、角色校验、独立 403 页面和按角色隔离的路由空间。
- 候选人登录、管理端角色登录和已接入后端接口的候选人注册页面。
- TanStack Vue Query 全局配置，以及职位、候选人、招聘流程、面试、Offer、入职和员工档案模块的查询与变更状态管理。
- 四角色共用消息中心、顶部未读数、单连接 SSE 实时刷新和断线重连提示。
- 四角色共用账户与安全设置，并在应用刷新时通过 `/auth/me` 校正会话。
- 页面内嵌的 AI 面试题、反馈摘要和离职风险参考；AI 不直接改变业务状态。
- 候选人中心、HR 招聘工作台、面试官任务工作台和管理员治理中心四套信息架构。
- 工作台、职位管理、候选人管理、招聘流程、面试工作区、Offer 管理、入职办理和员工档案页面。
- Vitest 单元测试、ESLint、Prettier 和 `vue-tsc` 检查链路。

当前仍需注意：

- 候选人注册页已接入后端 `/auth/register`，注册成功后写入候选人会话并进入候选人中心。
- 候选人验证码登录只完成界面，当前后端登录接口仅支持账号密码。
- 候选人中心已接入公开职位详情和本人资料、简历、投递详情、面试、Offer、入职接口；列表统一使用 `pageNum/pageSize`，任何 401/403 都不会回退到演示数据。
- 面试官与 HR 复用面试工作区视图，但任务接口和页面操作按当前角色限制；面试官首页预览仍使用 `/interviews/interviewer/me`。
- 管理员仅负责用户、角色、字典、审计和系统状态；治理接口未完成的页面明确显示待接入，不使用招聘业务数据占位。
- 职位、候选人、招聘流程、面试、Offer、入职和员工档案已有后端 Controller/API 对接；真实接口失败时只能由用户主动切换到明确标识的演示数据。
- 候选人管理支持 HR 新增、详情和编辑；职位详情可读取该职位投递，职位跳转会把 `jobId` 同步到招聘流程筛选。
- Offer 发送、撤回和完成入职动作不发送无意义请求体；候选人接受/拒绝 Offer 和完成入职均保留人工确认。
- 消息会话由 `applicationId` 创建，投递详情可联系对方；消息页支持会话、消息分页和标记已读。
- 候选人求职资料与登录账户资料分开维护，避免把 `Candidate` 字段混入 `SysUser` 设置。
- AI 原始响应不使用 `Result<T>`，由前端 API 层做运行时归一化；当前 AI 网关鉴权与角色校验仍需后端补强，不能仅依赖前端路由控制。
- `/hr/ai-approvals` 只保留无导航入口的能力边界说明，不宣称存在生产审批队列。
- 知识库尚无可用网关/统一响应契约，本轮不提供知识库页面。

新增业务功能时优先沿用现有边界：

- 路由页面放在 `views/<domain>/`。
- 可复用业务组件放在 `components/<domain>/`。
- 页面状态与交互逻辑放在 `composables/`。
- 静态文案、菜单和选项配置放在 `config/`。
- HTTP 和响应适配只放在 `api/`。
- 跨模块类型放在 `types/`。
- 只有出现多个完整业务模块后，才考虑新增 `features/`，不要提前迁移现有结构。

## 4. 应用启动流程

入口文件为 `src/main.ts`，启动顺序如下：

1. 引入 Element Plus 默认样式和 Recruit Smart 全局样式。
2. 创建 Vue 应用和 Pinia。
3. 注册 TanStack Vue Query。
4. 注册 Vue Router。
5. 注册 Element Plus。
6. 挂载到 `#app`；已有 Token 时由 `App.vue` 调用 `/auth/me` 校正当前用户信息。

`src/App.vue` 承载会话校正和根级 `<RouterView />`，登录页、注册页和登录后的应用框架由路由决定。

## 5. 路由与角色权限

当前路由表：

| 路径 | 页面 | 允许角色 | 状态 |
|---|---|---|---|
| `/login` | 登录页 | 公共 | 已实现 |
| `/register` | 候选人注册页 | 公共 | 已接入 `/auth/register` |
| `/` | 角色入口 | 全部已登录角色 | 按角色跳转到专属首页 |
| `/candidate/*` | 候选人中心 | `CANDIDATE` | 公开职位及本人资料、简历、投递、面试、Offer、入职 |
| `/hr/*` | 招聘管理工作台 | `HR` | 招聘主流程管理模块 |
| `/interviewer/*` | 面试任务工作台 | `INTERVIEWER` | 仅本人被分配的面试任务 |
| `/admin/*` | 系统治理中心 | `ADMIN` | 用户、角色、字典、审计和系统状态；接口待接入 |
| `/<role>/messages` | 消息中心 | 对应工作区角色 | 会话分页、消息分页、未读数、已读和 SSE |
| `/<role>/settings` | 账户与安全 | 对应工作区角色 | `/auth/me`、个人资料和密码修改 |
| `/forbidden` | 无权限页 | 全部已登录角色 | 保留会话并解释角色边界 |

旧的 `/jobs`、`/candidates`、`/pipeline`、`/interviews`、`/offers`、`/onboardings`、`/employees` 和 `/ai-approvals` 仅作为 HR 历史链接跳转；非 HR 访问目标路由时进入 403。

全局守卫行为：

1. `meta.public` 路由直接放行。
2. 未登录用户访问受保护路由时跳转 `/login`，并通过 `redirect` 保存目标地址。
3. `/` 根据当前角色跳转到 `ROLE_WORKSPACES` 中定义的专属首页。
4. 已登录但角色不匹配时进入 `/forbidden`，不加载目标页面，也不发起目标业务请求。
5. 导航、搜索范围和默认首页统一读取 `src/config/roleAccess.ts`，不再分别维护权限表。

前端权限只改善用户体验，不能代替后端鉴权。后端仍必须校验 Token、角色和数据归属；尤其是候选人详情类接口必须继续校验记录是否属于当前账号。

## 6. 登录与会话

登录请求位于 `src/api/auth.ts`：

```text
POST /api/auth/login
POST /api/auth/register
GET  /api/auth/me
PUT  /api/users/me
PUT  /api/users/me/password
```

开发环境中，Vite 会将 `/api` 代理到本地后端网关；注册与登录都通过统一 Axios 客户端提交。

登录和注册响应由 API 层转换为统一的 `LoginPayload`。仅接受以下角色：

- `ADMIN`
- `HR`
- `INTERVIEWER`
- `CANDIDATE`

后端返回空角色或未知角色时，前端抛出 `ApiError(403)`，禁止默认授予 HR 权限。

会话由 `src/stores/session.ts` 管理，当前使用 `sessionStorage`：

```text
recruit-smart-token
recruit-smart-token-type
recruit-smart-user
```

Axios 请求拦截器自动添加：

```http
Authorization: <tokenType> <token>
```

收到 HTTP `401` 时会清理本地会话。当前响应拦截器只负责清理，不直接执行路由跳转；需要跳转时应在统一认证失效流程中扩展，不能在业务页面重复处理。

## 7. API 与响应适配

所有 REST 请求统一使用 `src/api/http.ts`，默认配置为：

```text
baseURL: /api
timeout: 15000ms
```

后端统一响应类型：

```ts
export interface Result<T> {
  code: number
  message: string
  data: T | null
}
```

`unwrapResult<T>` 将 `code` 为 `0` 或 `200` 视为成功，其他业务码转换为 `ApiError`。更新、发布和关闭等接口成功时返回 `data: null`，由 `unwrapVoidResult` 单独处理。页面组件不得直接处理 Axios 原始响应壳。

前端分页内部类型用于统一页面消费格式：

```ts
export interface PagedData<T> {
  items: T[]
  page: number
  pageSize: number
  total: number
}
```

职位、候选人、投递、面试、Offer、入职、员工和消息列表统一把后端 `PageResult<T> { total, records }` 转为 `PagedData<T>`，请求统一发送 `pageNum/pageSize`。页面不得用固定 `pageSize=100` 伪造前端分页。

面试工作区使用以下已核对契约：

```text
GET  /interviews/tasks
GET  /interviews/{id}/workspace
PUT  /interviews/{id}/feedback/draft
POST /interviews/{id}/feedback
POST /ai/interview-questions
POST /ai/feedback-summary
```

`InterviewFeedback.comment` 与 `InterviewFeedback.aiSummary` 必须保持独立。提交面试反馈只记录面试官评分与建议，不直接改变投递、Offer 或录用状态。

Offer 管理使用以下已核对契约：

```text
GET  /offers
GET  /offers/{id}
POST /offers
PUT  /offers/{id}
PUT  /offers/{id}/send
PUT  /offers/{id}/revoke
```

HR 只能编辑草稿、发送草稿和撤回已发送 Offer。`ACCEPTED` 与 `REJECTED` 必须来自候选人动作或后端业务流程，前端不得提供 HR 代操作入口。

入职与员工档案使用以下已核对契约：

```text
GET  /onboarding
GET  /onboarding/{id}
PUT  /onboarding/{id}/approve-materials
PUT  /onboarding/{id}/reject-materials
PUT  /onboarding/{id}/complete
PUT  /onboarding/{id}/cancel
GET  /employees
GET  /employees/{id}
POST /ai/turnover-risk
```

`PUT /onboarding/{id}/complete` 必须由 HR 二次确认，并由后端在同一事务中把入职状态改为 `ONBOARDED`、创建唯一员工档案。员工状态仍通过业务接口和人工操作控制；离职风险只作参考，不得自动触发转正、调岗或离职。

## 8. 状态管理边界

### Pinia

当前 Pinia 只存储登录会话。后续可以存储跨页面、纯客户端且需要持久存在的 UI 状态，但不得把候选人列表、职位列表或分页结果复制进 Pinia。

### TanStack Vue Query

Vue Query 已在 `src/main.ts` 注册，默认策略为：

- 查询数据 `30s` 内视为新鲜。
- 查询失败重试 `1` 次。
- Mutation 不自动重试。
- 窗口重新聚焦时不自动刷新。

职位、候选人、招聘流程、面试、Offer、入职和员工档案分别通过对应 composable 管理服务端状态。各模块的 Mutation 成功后统一失效列表与详情缓存，不得把服务端列表复制到 Pinia。

## 9. 消息 SSE、AI 与人工决策

`src/api/messages.ts` 使用 `@microsoft/fetch-event-source` 连接 `GET /api/messages/stream`。工作区外壳持有唯一 SSE 连接，消息页复用共享状态和 Vue Query 缓存，避免进入消息页后重复连接。连接会携带 JWT，页面卸载时通过 `AbortController` 关闭；断线时显示提示，并由客户端退避重连。

消息 REST 契约包括会话创建/分页、消息分页/发送、标记已读和未读数。投递上下文先通过 `applicationId` 创建或取得会话，再携带 `conversationId` 打开消息页。

AI 接口返回原始 DTO，不遵循业务服务的 `Result<T>`。`src/api/ai.ts` 和 `src/api/interviews.ts` 必须归一化数组、枚举、分数和空值后再交给页面。面试题请求会补充岗位、候选人、简历和经历上下文；反馈摘要不得覆盖面试官原始评分和评价；离职风险不得修改员工状态。

当前 Gateway 已配置 `/api/ai/**` 路由，但尚未确认生产级鉴权、角色校验和调用审计。前端携带 JWT 和限制入口仅用于配合后端，不能作为安全边界。改变候选人阶段、发送 Offer、拒绝候选人、安排面试或转员工档案等操作，必须由业务服务执行，并经过用户明确确认。

## 10. 样式与设计实现

样式入口为 `src/styles/index.scss`：

- `tokens.scss` 定义颜色、间距、圆角、阴影、动效和布局尺寸。
- `element-plus.scss` 将 Element Plus 映射到 Recruit Smart Token。
- `login.scss` 和 `register.scss` 只服务认证页面。

当前关键约束：

- 主色为 `#3183D8`。
- 桌面端最低完整宽度为 `1280px`，全局 `html` 设置了 `min-width: 1280px`。
- 页面高度优先使用 `100dvh`，避免浏览器工具栏造成溢出。
- 动效必须适配 `prefers-reduced-motion`。
- 新增视觉值前必须先检查 `design.md` 和 `tokens.scss`。
- Element Plus 只提供控件能力，不决定产品视觉。

- `interviews.spec.ts`：面试任务适配、组合筛选、评分换算、草稿和反馈锁定规则。
- `offers.spec.ts`：Offer 分页适配、组合筛选、发送校验、草稿编辑和人工状态变更规则。
- `onboardings.spec.ts`：入职分页适配、材料审核、退回补充和人工确认建档规则。
- `employees.spec.ts`：员工档案分页适配、组合筛选和 AI 风险空值语义。
- `candidate-portal.spec.ts`：候选人公开职位和本人投递的分页参数与角色边界。
- `messages.spec.ts`：会话/消息分页和标记已读动作契约。

- 路由角色守卫。
- `unwrapResult<T>` 成功与错误映射。
- 登录表单提交和错误状态。

## 13. 新功能落地规则

1. 开发前读取 `design.md` 和本文档。
2. 先确认对应后端接口和字段，不在页面中猜测契约。
3. API 调用放在 `api/`，响应适配不得散落在组件中。
4. 服务端状态优先使用 Vue Query，纯客户端跨页状态才使用 Pinia。
5. 路由通过 `meta.roles` 声明角色，侧栏与路由守卫保持一致。
6. 页面组件只负责组合，复杂交互放入 `composables/`。
7. 表单必须覆盖字段错误、表单错误和重复提交保护。
8. 页面必须考虑加载、空数据、错误、无权限和成功状态。
9. AI 建议必须展示来源，并保留人工确认入口。
10. 完成后运行类型检查、Lint、测试和生产构建。

## 14. 禁止事项

- 禁止绕过 `/api` 和统一 HTTP 客户端直接访问业务服务。
- 禁止页面组件自行读写 JWT。
- 禁止把服务端列表缓存复制到 Pinia。
- 禁止把尚未接入的依赖描述成已完成业务能力。
- 禁止未知角色回退为 HR 或其他高权限角色。
- 禁止 AI 未经用户确认直接改变招聘业务状态。
- 禁止为了 Element Plus 默认样式偏离 `design.md`。
- 禁止在未更新本文档的情况下更换核心框架、路由、状态管理或认证策略。
